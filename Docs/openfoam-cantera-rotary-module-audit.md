# OpenFOAM + Cantera Rotary Module Audit

Status: active design audit
Last updated: 2026-03-10
Parent roadmap: `Docs/main-development-plan.md`

This document adapts the generic "high-cost OpenFOAM + Cantera engine-cycle module"
checklist to the current `larrak2` stack.

It is not a second roadmap. Its job is to answer one narrower question:

How do we stand up simulation truth runs for NN training without losing:

1. rotary-valve geometry fidelity, or
2. non-sinusoidal piston motion, velocity, and acceleration?

## 1. Current Repo Reality

The codebase is not starting from zero, but it is also not yet a live
OpenFOAM <-> Cantera coupled cycle solver.

What already exists:

- Rotary-valve geometry generation for OpenFOAM STL injection exists in
  `src/larrak2/geometry/generate_stl.py`.
- Rotary-port effective-area scheduling exists in
  `src/larrak2/thermo/port_flow.py`.
- Valve timing is already derived from the candidate-specific volume trace in
  `src/larrak2/thermo/valve_timing.py`.
- The high-fidelity training/orchestration path already assumes an OpenFOAM
  truth loop and NN surrogate promotion.
- The tracked OpenFOAM template at
  `openfoam_templates/opposed_piston_rotary_valve_sliding_case` is currently a
  repo-contained nonreacting tracer/scavenging case, not a chemistry-coupled
  engine CFD stack.

What is still missing:

- No Cantera dependency or in-repo Cantera integration contract exists yet.
- The thermo volume law is still generated by `_phase_driven_volume(...)` in
  `src/larrak2/thermo/two_zone.py`, which supports asymmetric phase durations
  but not a fully arbitrary displacement/velocity/acceleration input trace.
- The current rotary-port flow model uses open/close angles plus a smooth ramp;
  it does not yet derive effective flow area directly from valve-window and
  port geometry overlap.

## 2. Modified Module Classes

The original A/B/C split is close, but for this repo one extra intermediate
class is useful.

| Class | Meaning | Fit for this repo now |
| --- | --- | --- |
| A | OpenFOAM-only reacting CFD | Too early; chemistry path is not the first blocker |
| B0 | OpenFOAM-first truth module for geometry, motion, scavenging, and charge preparation; chemistry handoff placeholders only | Immediate target |
| B | OpenFOAM + Cantera handoff module with explicit reduced-state transfer | Next target after B0 |
| C | Cantera-led reduced cycle with selective OpenFOAM support | Later option, not the current training bottleneck |

Recommended target sequence:

1. Build `B0` to generate authoritative motion- and geometry-faithful truth
   data for NN training.
2. Promote to `B` only after the state-reduction and motion contracts are
   stable.

That keeps the current optimization system moving without forcing a premature
chemistry coupling rewrite.

## 3. Required Changes to the Generic Checklist

These are the key modifications to the generic two-stroke pick list.

### 3.1 Rotary-valve geometry must be treated as a preserved invariant

Replace generic "valve / port architecture" wording with:

- Rotary-valve solid geometry revision
- Rotary-valve window geometry revision
- Port-throat geometry revision
- Valve-to-port phasing convention
- Interface/clearance geometry used by the OpenFOAM mesh
- Geometry simplifications explicitly recorded

For this stack, a timing law alone is not enough. The optimization/training
path must preserve the geometry that produces the timing and area schedule.

### 3.2 Piston motion must be a full kinematic contract, not only a displacement law

Replace "piston displacement law vs cycle coordinate" with:

- piston displacement `x(theta)`
- piston velocity `dx/dtheta` and `dx/dt`
- piston acceleration `d2x/dtheta2` and `d2x/dt2`
- optional jerk metric for gating
- cycle closure check
- TDC/BDC event detection method robust to dwell/plateau motion

The current solver derives timing anchors from the volume trace, which is the
right direction, but the source motion trace must become first-class rather
than being implied by phase durations.

### 3.3 Rotary-port exposure should become geometry-derived

The present `rotary_port_area_schedule(...)` is acceptable as a reduced model,
but the high-cost truth path should ultimately use:

- geometric overlap area between the valve window and port opening, or
- a precomputed angle-to-area table derived from the actual valve geometry.

Open/close angles remain useful, but they should be outputs of geometry and
motion, not the sole authority.

### 3.4 Chemistry handoff must carry motion context

If Cantera is introduced later, the handoff state should include not just
pressure, temperature, and composition, but also:

- current cycle coordinate
- local or reduced motion-state markers
- charge-preparation completion metrics tied to the actual motion law

This matters because non-sinusoidal motion changes compression-rate history and
therefore ignition-relevant thermodynamic trajectories.

## 4. Repo-Specific Readiness Checklist

Use `Known`, `Estimated`, `Needs measurement`, `Needs modeling assumption`, or
`Not yet defined`.

### A. System definition

- Cycle type fixed as rotary-valve two-stroke
- Fueling mode fixed for the first truth campaign
- Ignition mode fixed for the first truth campaign
- Whether the first truth campaign is nonreacting, reacting, or mixed
- Whether the first campaign is for scavenging only, charge prep, or full cycle
- Single-cylinder vs full hardware representation

### B. Geometry definition

- Rotary-valve CAD/STL ownership and revision tag
- Valve drum OD/ID/length
- Valve window shape, arc, axial extent, and phasing reference
- Intake and exhaust port throat geometry
- Chamber geometry and piston crown geometry
- Leakage paths and clearances included vs omitted
- Mesh-ready geometry simplification rules

### C. Motion definition

- Canonical motion trace source file or generator
- `x(theta)` sampled over one closed cycle
- `v(theta)` and `a(theta)` available from the same source
- Volume trace generated from motion, not only from duration heuristics
- TDC/BDC detection validated for asymmetric motion laws
- Rotary-valve angular position law and angular velocity law
- Port exposure timing derived from geometry + motion

### D. OpenFOAM truth module definition

- Dynamic-mesh strategy selected for piston motion
- Rotary-valve motion strategy selected
- Static/tracer, spray, and reacting configurations separated cleanly
- Required training outputs fixed in advance
- Mesh quality thresholds defined near minimum volume and port edges
- Run-cost tiers defined from day one

### E. Chemistry / Cantera readiness

- Mechanism source selected
- Fuel surrogate selected
- Reduced-state handoff definition chosen
- Zone reduction rule chosen
- Ignition-only vs full burn chemistry scope fixed
- Cantera YAML validation path defined

### F. Handoff-state contract

- End-of-scavenging state bundle defined
- End-of-charge-preparation state bundle defined
- Ignition-start state bundle defined
- Reduction rule documented
- Provenance manifest for every truth case

### G. NN-training contract

- OpenFOAM truth labels fixed before campaign launch
- Training inputs include geometry and motion descriptors, not only scalar BCs
- Dataset coverage metadata includes motion-law family and geometry revision
- Authority bundle promotion rules updated for the new fields

## 5. Additional Mandatory Fields for This Stack

These fields should be added to any future truth-case manifest, dataset row, or
handoff bundle.

### 5.1 Motion fields

- `motion_profile_id`
- `motion_profile_revision`
- `volume_trace_source`
- `x_theta_samples`
- `v_theta_samples`
- `a_theta_samples`
- `tdc_deg`
- `bdc_deg`
- `motion_cycle_closure_error`

### 5.2 Rotary-geometry fields

- `rotary_valve_geometry_id`
- `rotary_valve_window_arc_deg`
- `rotary_valve_window_length_frac`
- `intake_port_geometry_id`
- `exhaust_port_geometry_id`
- `angle_area_table_id` or equivalent geometry-derived exposure source

### 5.3 Truth outputs for NN training

At minimum:

- trapped mass
- scavenging efficiency
- residual fraction
- trapped O2 mass
- delivery ratio
- trapping efficiency
- charge-temperature descriptor
- motion-sensitive timing markers

If chemistry is later added:

- reduced ignition-delay targets
- zone-reduced composition descriptors
- heat-release anchor points

## 7. Recommended Build Order

### Phase 1: Finish the `B0` truth path

- Keep OpenFOAM focused on scavenging, trapped-mass, residual, tracer mixing,
  and motion/geometry sensitivity.
- Make motion law explicit in the case-generation contract.
- Preserve rotary geometry as a versioned input, not a hidden assumption.

### Phase 2: Replace heuristic motion inputs with a kinematic contract

- Introduce a canonical motion-profile artifact or schema.
- Generate volume from the motion profile.
- Re-derive valve timing anchors from the resulting volume trace.
- Export motion descriptors into truth datasets and NN features.

### Phase 3: Upgrade rotary-port exposure from timing-based to geometry-based

- Produce angle-to-area tables from the actual valve/port geometry.
- Use those tables in both reduced thermo and OpenFOAM case generation.
- Keep timing degrees as diagnostics, not the primary authority.

### Phase 4: Add Cantera only after the reduction contract is stable

- Decide zone reduction strategy.
- Define the exact handoff bundle.
- Validate one chemistry path first: ignition-delay / pre-ignition / reduced
  heat-release anchoring.

## 8. Direct Conclusion

For this repo, the right near-term target is not "generic OpenFOAM + Cantera
cycle simulation." It is:

`OpenFOAM-first truth generation that preserves rotary geometry and arbitrary
motion laws well enough to train the NN and feed the existing optimization
stack.`

Cantera becomes the next layer only after the motion and geometry contracts are
authoritative.
